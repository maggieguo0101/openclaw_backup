# AIBD 签约&上团单 × OpenClaw 企微自动化方案

## 一、背景与目标

AIBD 外呼筛选出意向商家后，需要通过销售侧完成签约和上团单。传统方式依赖真人销售逐一在企微群内跟进，效率低、覆盖量有限。

本方案目标：通过 OpenClaw 附身在企微账号「王晓慧」上，在 2000 个商家企微群内自动完成签约引导和上团单流程，实现销售流程的 AI 自动化。

---

## 二、整体架构

```
后台任务系统
    │ 批量触发指令
    ▼
OpenClaw（核心 Agent）
    │ 调用 Skill
    ├─→ 商家认领状态查询 Skill（MVP 阶段 Mock）
    ├─→ 签约 Skill（发小程序链接、状态追踪）
    └─→ 上团单 Skill（签约完成后提交）
    │
    │ 消息发送
    ▼
深海平台（消息转发层）
    │
    ▼
王晓慧企微账号 → 各商家企微群
```

**关键设计原则：**
- 每个商家群 = 独立 Session，上下文严格隔离，互不干扰
- 王晓慧是专用企微机器人账号，通过深海平台 API 收发消息
- OpenClaw 不直连企微，所有消息经深海平台中转

## 三、核心业务流程

### 3.1 主流程图

```
后台批量触发 / 商家主动发消息
         │
         ▼
  [OpenClaw 接收群消息]
         │
         ▼
  调用「认领状态查询 Skill」
  查询该商家是否已认领
         │
    ┌────┴────┐
    │         │
  未认领     已认领
    │         │
    ▼         ▼
发送开场白   回复说明情况
+ 签约小程序  请商家发材料
链接         （营业执照+身份证）
    │         │
    ▼         ▼
等待商家    收到材料后
在小程序内  通知真人销售
完成签约    代操作跟进
    │
    ▼
签约审批完成（Skill 回调/轮询）
    │
    ▼
调用「上团单 Skill」提交团单
    │
    ▼
群内通知商家：上团单已提交
```

### 3.2 超时提醒机制

| 场景 | 触发条件 | 动作 |
|------|----------|------|
| 商家未回复 | 发送开场白后 3 小时无回复 | 自动发送一次提醒 |
| 小程序未完成 | 发送链接后 3 小时未签约 | 再次提醒+重发链接 |
| 材料未提交（已认领） | 请材料后 3 小时无回复 | 提醒一次，再通知真人销售 |
| 签约审批超时 | 提交后 3 小时未审批完成 | 通知真人销售跟进 |

**夜间静默规则：**
- 提醒触发时间若在 **22:00 ~ 08:00**，自动延迟至次日 08:00 发送
- 计时器不受影响，仅发送动作被推迟

## 四、多群隔离设计

### 4.1 Session 隔离方案

OpenClaw 为每个企微群维护独立 Session：

```
群 ID: group_abc123  →  Session: aibd:group_abc123
群 ID: group_def456  →  Session: aibd:group_def456
...（2000个群，2000个独立Session）
```

- 每个 Session 存储该商家群的完整对话上下文、当前流程状态、商家信息
- Session 间完全隔离，不共享任何上下文
- Session 状态持久化，重启后可恢复

### 4.2 流程状态机

每个 Session 维护一个状态机，防止流程错乱：

```
IDLE → GREETING_SENT → CHECKING_STATUS
     → UNCLAIMED_LINK_SENT → WAITING_SIGN → SIGN_COMPLETE → ORDER_SUBMITTED
     → CLAIMED_MATERIAL_REQUESTED → MATERIAL_RECEIVED → HUMAN_NOTIFIED
```

### 4.3 是否需要为每个商家部署独立 OpenClaw 实例？

**结论：不需要。** 原因如下：

| 维度 | 独立实例 | 共享实例（推荐） |
|------|----------|-----------------|
| 上下文隔离 | 天然隔离 | Session 级隔离，同样安全 |
| 资源消耗 | 2000 × 资源 | 1 个实例即可 |
| 运维复杂度 | 极高 | 低 |
| 话术差异化 | 方便 | 通过 Prompt 模板 + 商家元数据实现 |
| 适用场景 | 不同业务线/不同账号 | 同一账号服务多商家 ✅ |

**例外情况**：如果未来需要多个企微账号（如多个王晓慧）或不同业务线，再考虑多实例部署。

## 五、需要开发的 Skill 清单

### Skill 1：商家认领状态查询

| 项目 | 内容 |
|------|------|
| 功能 | 根据商家 ID/企微群 ID 查询是否已认领 |
| MVP | Mock 数据，返回 `{claimed: true/false, merchantName, merchantId}` |
| 正式 | 接入 AIBD 商家信息接口 |
| 输入 | 商家 ID 或企微群 ID |
| 输出 | 认领状态、商家基本信息 |

### Skill 2：签约小程序链接生成

| 项目 | 内容 |
|------|------|
| 功能 | 根据商家信息生成专属签约小程序链接 |
| 输入 | 商家 ID |
| 输出 | 签约小程序链接、链接有效期 |

### Skill 3：签约状态查询

| 项目 | 内容 |
|------|------|
| 功能 | 查询商家签约审批进度 |
| 输入 | 商家 ID / 签约单 ID |
| 输出 | 签约状态（待审批/审批中/已完成/已拒绝） |

### Skill 4：上团单提交

| 项目 | 内容 |
|------|------|
| 功能 | 签约完成后自动提交上团单 |
| 输入 | 商家 ID、行业类型、团单信息 |
| 输出 | 团单 ID、提交状态 |

**团单字段说明：**

- **必填（所有行业通用）：** 团单名称、门市价、团购价
- **行业差异字段：** 按行业加载对应字段模板（餐饮/酒店/休闲娱乐等各不相同）
- **字段收集方式：** OpenClaw 在群内通过多轮对话引导商家逐步提供，收集完成后校验必填项，再提交团单

> 行业字段模板需由业务侧维护，Skill 根据商家行业动态加载对应模板。

### Skill 5：真人销售通知

| 项目 | 内容 |
|------|------|
| 功能 | 已认领商家材料收集完成后，通过大象通知对应真人销售 |
| 输入 | 商家 ID、材料列表（图片/文件链接）、群 ID、销售 misId |
| 输出 | 大象消息发送状态 |
| 通知内容 | 商家名称、群链接、已收材料清单，提示销售去代操作 |

## 六、深海平台接入方案

深海平台承担 OpenClaw ↔ 企微 的消息中转：

**消息接收（企微群 → OpenClaw）：**
1. 商家在企微群发消息
2. 深海平台接收企微 Webhook 推送
3. 深海平台将消息转发至 OpenClaw 对应 Session
4. OpenClaw 处理后生成回复

**消息发送（OpenClaw → 企微群）：**
1. OpenClaw 生成回复内容
2. 调用深海平台发送接口
3. 深海平台通过王晓慧账号发送至对应企微群

**需要与深海平台对齐的接口：**
- `POST /send` - 发送消息（文本/图片/小程序卡片）
- `GET /groups` - 获取王晓慧所在群列表
- Webhook 回调格式 - 接收群内消息

## 八、风险与注意事项

| 风险 | 说明 | 应对 |
|------|------|------|
| 企微账号限流 | 机器人账号高频发消息可能触发限流 | 深海平台控制发送频率，错峰发送 |
| 商家隐私 | 2000个商家信息在同一实例，需严格隔离 | Session 隔离 + 数据访问控制 |
| 流程中断恢复 | OpenClaw 重启后能否恢复进行中的流程 | Session 状态持久化到数据库 |
| 误操作风险 | 把 A 商家的材料发给 B 商家 | 严格的群 ID ↔ 商家 ID 绑定校验 |
| 人工介入时机 | 复杂问题 Agent 处理不了 | 明确兜底规则，及时转人工 |
